# 班級缺考登錄系統

> 供各班老師在考試後回報缺考座號，資料即時同步到雲端 Google 試算表；管理處可查看、彙整並匯出所有班級的回報狀況。

- **最後更新**：_請接手者自行填寫_
- **維護者**：_請接手者自行填寫_

---

## 1. 這是什麼

一個「班級缺考登錄系統」的網頁工具。老師在考試後回報缺考座號，資料即時同步到雲端 Google 試算表；管理員（管理處）可查看、彙整、匯出全校回報狀況。

**特色**

- 整個前端只有一個檔案 `index.html`（HTML + CSS + JavaScript），無須自架伺服器，直接放在 GitHub Pages 就能運作。
- 資料儲存與權限驗證由 **Google Apps Script + Google 試算表** 負責。
- 以手機操作為主，座號用點選方式，直觀快速。

---

## 2. 系統架構

```
[老師的手機 / 瀏覽器]
      │  開啟 GitHub Pages 上的 index.html
      ▼
[index.html 前端]  ←── 輸入密碼解密後，取得後端網址
      │  透過 fetch 呼叫 Google Apps Script（帶 token 驗證）
      ▼
[Google Apps Script 後端]
      │  讀寫
      ▼
[Google 試算表（資料庫）]
```

| 層級 | 角色 |
| --- | --- |
| 前端（`index.html`） | 畫面、座號點選、登入解密 |
| 後端（Apps Script） | 真正的資料存取與權限把關 |
| 資料庫（試算表） | 所有缺考紀錄儲存於此 |

> ⚠️ **重要觀念**：前端沒有真正的秘密，安全防線在後端 Apps Script。詳見〈8. 安全性說明〉。

---

## 3. 使用方式

### 一般老師（回報缺考）

1. 開啟系統網址，輸入「班級存取密碼」後登入。
2. 選擇 **年段 → 年級 → 班級**。
3. 點選缺考的座號（可多選）；若全班到齊，按「全班到齊」，或不點任何座號直接提交（系統視為全班到齊）。
4. 按「確認提交登錄」→ 再按「確認送出」，即完成回報。

### 管理員（查看 / 彙整 / 匯出）

1. 登入畫面勾選「以管理員身分登入」，輸入「管理員專屬密碼」。
2. 登入後底部會多出「紀錄」分頁，可以：
   - 依 **日期 / 年段 / 年級** 篩選
   - 查看缺考人數與名單
   - **複製全部**：把當日彙整貼到 LINE / 通訊軟體
   - **匯出報表**：下載 Excel（`.xls`）
   - **完成回報班級一覽**：檢視哪些班還沒回報（🟥 未回報、🟩 已回報）
   - 刪除單筆、或清除全部紀錄（清除為雲端全刪，**無法復原**）

## 4. 登入與加密機制（接手者務必理解）

後端網址**沒有**明文寫在 `index.html`，而是加密後存成兩段字串，必須輸入正確密碼才解得開。

程式裡的兩個變數：

| 變數 | 用哪組密碼加密 | 解開後得到 |
| --- | --- | --- |
| `ENCRYPTED_DATA` | 一般密碼 | 後端網址 |
| `ENCRYPTED_ADMIN` | 管理員密碼 | 字串 `ADMIN_VERIFIED:` + 一般密碼 |

**登入流程**

- 老師輸入一般密碼 → 解開 `ENCRYPTED_DATA` → 取得後端網址 → 連線驗證。
- 管理員輸入管理員密碼 → 解開 `ENCRYPTED_ADMIN` → 取出裡面的一般密碼 → 再解開 `ENCRYPTED_DATA` → 取得後端網址；同時把 `isAdmin` 設為 `true`。

**加密規格**（自行重新產生密文時需依此格式）

- 金鑰推導：PBKDF2、SHA-256、迭代 **100000** 次
- 加密演算法：AES-GCM、256 bit
- 密文位元組結構（base64 解碼後）：

  | 位元組範圍 | 內容 |
  | --- | --- |
  | `0 – 15` | salt（16 bytes） |
  | `16 – 27` | iv（12 bytes） |
  | `28 – 43` | authentication tag（16 bytes） |
  | `44 –` | 實際密文（ciphertext） |

- 前端解密程式在 `index.html` 的 `decryptData()` 函式，可對照。

> ⚠️ 這份 repo **不含**產生密文的加密工具。若要更換後端網址或變更密碼，必須用相同規格重新加密（見〈6. 常見維護工作〉）。請務必把加密用的小工具／腳本另外保存好，**遺失會很麻煩**。

## 5. 部署（放到 GitHub Pages）

1. 把 `index.html` 放進 GitHub repo。
2. repo → **Settings → Pages → Source** 選 `main` 分支、根目錄。
3. 等它產生網址（形如 `https://帳號.github.io/repo名/`）即可使用。
4. `README.md` 放在 repo 根目錄，GitHub 會自動渲染在首頁。

**建立第二套獨立系統**（不同處室 / 不同試算表）

- 複製一份 `index.html` 到另一個 repo。
- 用該系統專屬的後端網址與密碼重新產生 `ENCRYPTED_DATA` / `ENCRYPTED_ADMIN`，替換進去即可。兩套資料互不干擾。

## 6. 常見維護工作

以下皆為修改 `index.html` 內容。

**(A) 更換後端網址**（換了新的 Apps Script 部署）
用「一般密碼」把新網址依〈4〉規格重新加密，取代 `ENCRYPTED_DATA`。若密碼也要變，連 `ENCRYPTED_ADMIN` 一起重做，並提醒使用者重新登入（舊 token 存在 `localStorage`）。

**(B) 修改密碼**
- 一般密碼變更 → 需同時重做 `ENCRYPTED_DATA` 與 `ENCRYPTED_ADMIN`（後者內含一般密碼）。
- 管理員密碼變更 → 只需重做 `ENCRYPTED_ADMIN`。

**(C) 更改每班座號上限**
找到 `var MAX_SEAT = 55`，改成需要的數字（目前為 55）。

**(D) 更改座號排列（一行幾格）**
CSS 中 `.seat-picker` 的 `grid-template-columns: repeat(10,1fr)`，把 `10` 改成想要的欄數。

**(E) 更改班級名冊（新增／刪除班級、改班級代號）**
- 修改 JavaScript 中的 `REPORT_ROSTER` 陣列（控制「完成回報一覽」與匯出排序的班級順序）。
- 登入畫面的下拉選單由 `updateClassOpts()` 函式產生，班級代號有變時這裡也要一併調整。

**(F) 調整顏色／字體對比**
CSS 最上方 `:root` 定義了所有顏色變數（`--text`、`--red` 等），改這裡會全站套用。

## 7. 資料結構與後端介面

**每筆紀錄的欄位**

| 欄位 | 說明 | 範例 |
| --- | --- | --- |
| `id` | 建立時的時間戳記（唯一識別） | `1783661132460` |
| `taskType` | 固定為 `"absence"` | `absence` |
| `className` | 班級全名 | `高中部三年甲班`、`國中部一年10班` |
| `seats` | 缺考座號陣列；空陣列代表全班到齊 | `[3, 12, 25]` |
| `allPresent` | 是否全班到齊 | `true` / `false` |
| `date` | 月/日 | `7/10` |
| `time` | 時:分 | `08:30` |

**前端呼叫後端的動作**（Apps Script 需支援）

| action | 參數 | 回傳 |
| --- | --- | --- |
| `getAll` | — | `{ records: [...] }` |
| `add` | `record` | `{ ok: true }` |
| `delete` | `id` | 成功即可 |
| `clear` | — | 成功即可 |
| （驗證失敗） | — | `{ error: "unauthorized" }` |

所有請求都會帶 `token` 參數（＝一般密碼），由後端驗證。

> 📌 **試算表小提醒**：Google 試算表會自動把看起來像日期／時間的字串轉格式。前端已用 `formatDateDisplay()`、`formatTimeDisplay()` 做容錯；若日後時間顯示異常，優先檢查這兩個函式與試算表欄位的儲存格格式。

## 8. 安全性說明（很重要）

這套系統對「校內缺考登記」這種低敏感度用途堪用，但要清楚它的極限：

1. **前端加密只能擋沒有密碼的路人。** 合法登入者一旦解鎖，後端網址與 token 就以明文存在瀏覽器記憶體與 `localStorage`（`cls_token` / `cls_role`），碰得到裝置的人就拿得到。
2. **管理員／一般角色目前只在瀏覽器端判斷（`isAdmin`）。** 送到後端的 token 兩者相同（都是一般密碼）。只要有人拿到一般密碼，理論上在開發者工具手動把 `isAdmin` 設為 `true`，就能看到／清除全部紀錄。若要真正保護管理員權限，須改成**後端用兩組不同 token 分辨角色**，讀取／刪除／清空要求管理員 token 才放行。
3. **公開 repo 的密文可離線破解。** `ENCRYPTED_DATA` / `ENCRYPTED_ADMIN` 任何人都下載得到，密碼務必**夠長、夠隨機**，避免簡單好猜的字串。
4. **token 目前用網址查詢字串（`GET ?token=...`）送出**，可能被記進伺服器日誌或瀏覽器歷史；改用 POST body 傳送較安全。

> ✅ **結論**：真正的權限控管要放在 Google Apps Script 後端。若要提升安全性，建議**優先處理第 2 點**（後端雙 token 角色驗證）。

## 9. 已知問題 / 給接手者的提醒

- `index.html` 開頭有一行 `window.onerror = function(){ return true; };`，會**吞掉所有 JavaScript 錯誤**。正式使用時避免使用者看到報錯是好事，但你除錯時會看不到任何錯誤訊息，建議先暫時註解掉再抓問題。
- 程式裡的 `sortRecordsForExport()` 目前**沒有被呼叫**（死程式碼），實際匯出用的是 `sortRecordsByClass()`。兩者排序主鍵不同，若覺得匯出順序不如預期，可留意這點。
- `renderRecords()` 對 `r.seats` 少數情況未做防呆；若後端傳回缺少 `seats` 欄位的資料可能導致該頁渲染失敗（且因上面的 `window.onerror` 會靜默無訊息）。建議加上 `Array.isArray(r.seats)` 判斷。
